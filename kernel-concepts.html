第一部分：内核 = 事件处理器 + 数据结构

内核本质上是一个被动的事件驱动系统：等待事件发生，然后操作数据结构来响应。

Handler1.1 系统调用处理 (System Call Handler)

用户程序通过 svc 指令触发同步异常，CPU 从 EL0 跳转到 EL1 的异常向量表。以下是 ARM64 上完整的系统调用入口链路：

完整调用链

用户态: svc #0 → vectors[] (entry.S:519) → el0t_64_sync (entry.S:598) → el0t_64_sync_handler (entry-common.c) → do_el0_svc (syscall.c:149) → el0_svc_common (syscall.c:73) → invoke_syscall (syscall.c:38) → sys_call_table[nr]

// Line 149 — SVC 入口
void do_el0_svc(struct pt_regs *regs)
{
    // regs[8] = 系统调用号 (x8 寄存器), sys_call_table 是函数指针数组
    el0_svc_common(regs, regs->regs[8], __NR_syscalls, sys_call_table);
}

// Line 73 — 公共处理逻辑
static void el0_svc_common(struct pt_regs *regs, int scno,
                           int sc_nr, const syscall_fn_t syscall_table[])
{
    regs->orig_x0 = regs->regs[0];
    regs->syscallno = scno;
    // ... ptrace/seccomp 钩子检查 ...
    invoke_syscall(regs, scno, sc_nr, syscall_table);  // Line 132
}

// Line 38 — 查表分发
static void invoke_syscall(struct pt_regs *regs, unsigned int scno,
                          unsigned int sc_nr,
                          const syscall_fn_t syscall_table[])
{
    if (scno < sc_nr) {
        syscall_fn_t syscall_fn;
        // array_index_nospec 防止 Spectre v1 越界读取
        syscall_fn = syscall_table[array_index_nospec(scno, sc_nr)];
        ret = __invoke_syscall(regs, syscall_fn);
    } else {
        ret = do_ni_syscall(regs, scno);  // 未实现的系统调用
    }
    syscall_set_return_value(current, regs, 0, ret);
}

Handler1.2 中断处理 (Interrupt Handler)

硬件中断通过 GIC (Generic Interrupt Controller) 分发到 CPU。ARM64 的 IRQ 向量最终调用 handle_arch_irq 函数指针，该指针在 GIC 初始化时被设置为 gic_handle_irq。

IRQ 调用链

GIC 硬件 → el0t_64_irq (entry.S) → handle_arch_irq (irq.c:100) → gic_handle_irq (irq-gic-v3.c:955) → generic_handle_domain_irq() → 设备 ISR

GIC v3 中断控制器

linux-src/drivers/irqchip/irq-gic-v3.c

// Line 955 — GIC 注册的顶层 IRQ 处理函数
static void __exception_irq_entry gic_handle_irq(struct pt_regs *regs)
{
    u32 irqnr = gic_read_iar();  // 从 GIC 读取中断号
    if (gic_irqnr_is_special(irqnr))
        return;                 // Spurious / EOI
    gic_complete_ack(irqnr);    // 向 GIC 确认中断
    generic_handle_domain_irq(gic_data.domain, irqnr);
}

Virtio-net 中断 → NAPI → TCP 收包链

virtnet_interrupt (virtio_net.c) → napi_schedule() → virtnet_poll (virtio_net.c:2838) → virtnet_receive() → napi_gro_receive() → tcp_v4_rcv() → sk_add_backlog() / sk_data_ready()

linux-src/drivers/net/virtio_net.c : 2838

static int virtnet_poll(struct napi_struct *napi, int budget)
{
    struct receive_queue *rq = container_of(napi, struct receive_queue, napi);
    virtnet_poll_cleantx(rq, budget);
    received = virtnet_receive(rq, budget, &xdp_xmit);
    if (received < budget)
        virtqueue_napi_complete(napi, rq->vq, received);
    return received;
}

NAPI 机制要点：中断只负责调度 NAPI poll，不在中断上下文中直接处理数据包。这避免了高流量下频繁中断导致的 "活锁"（livelock）。poll 函数以 batch 方式处理多个包，完成后才重新开启中断。

Handler1.3 异常处理 (Exception Handler)

缺页异常 (Page Fault) 处理流程

MMU 触发 Permission Fault → el0t_64_sync (ESR 标识为 fault) → do_page_fault (fault.c:521) → handle_mm_fault() → handle_pte_fault() → do_anonymous_page() 或 wp_page_copy()

linux-src/arch/arm64/mm/fault.c : 521

static int __kprobes do_page_fault(unsigned long far,
                                   unsigned long esr,
                                   struct pt_regs *regs)
{
    struct mm_struct *mm = current->mm;
    unsigned long addr = untagged_addr(far);

    // 根据 ESR 判断故障类型
    if (is_el0_instruction_abort(esr))   vm_flags = VM_EXEC;
    else if (is_write_abort(esr))        vm_flags = VM_WRITE;
    else                                 vm_flags = VM_READ | VM_WRITE;

    // 查找 VMA，验证访问权限，然后调用 handle_mm_fault()
    vma = lock_mm_and_find_vma(mm, addr, regs);
    fault = handle_mm_fault(vma, addr, mm_flags, regs);
}

}

COW 时序图

fork() 之后:
  Parent PTE ──→ [Page A] ←── Child PTE   (两者都只读)

Parent 写入 Page A:
  1. MMU 触发 Write Fault
  2. wp_page_copy() 分配 [Page B]
  3. 拷贝 A → B
  4. Parent PTE ──→ [Page B] (可写)
     Child  PTE ──→ [Page A] (仍只读)

第二部分：内核 = 裁判 + 管家 + 翻译官

从功能角色理解内核的三个核心职责。

Referee2.1 裁判 — 强制执行规则与公平

2.1.1 CPU 调度 — CFS (Completely Fair Scheduler)

CFS 的核心思想：每个进程获得与其权重成比例的 CPU 时间。通过维护一个 "虚拟运行时间" (vruntime) 来实现公平性。

};

schedule() 选任务的流程

linux-src/kernel/sched/core.c : 6549 (__schedule)

static void __sched notrace __schedule(int sched_mode)
{
    struct task_struct *prev, *next;
    struct rq *rq = cpu_rq(smp_processor_id());
    prev = rq->curr;

    local_irq_disable();
    rq_lock(rq, &rf);
    update_rq_clock(rq);

    // 如果 prev 正在阻塞 (非抢占)，将其移出运行队列
    if (!preempt && prev_state)
        deactivate_task(rq, prev, flags);

    // ★ 选择下一个任务
    next = pick_next_task(rq, prev, &rf);
    //   → fair_sched_class.pick_next_task()
    //   → pick_task_fair() → 从红黑树取最左节点

    context_switch(rq, prev, next, &rf);
}

CFS 公平性原理

vruntime 计算：
  delta = now - se.exec_start          // 实际运行时间增量
  se.vruntime += delta * NICE_0_LOAD / se.load.weight

  → 权重越高 (nice 值越低)，vruntime 增长越慢
  → vruntime 最小的进程优先被调度

红黑树示意：
           [vr=50ms]
          /         \
     [vr=30ms]    [vr=70ms]
      /
  [vr=10ms] ← pick_next 总是选这里 (最左)

新进程插入：
  se.vruntime = cfs_rq.min_vruntime - sched_latency/2
  → 保证新进程能快速获得 CPU，但不插队太多

2.1.2 内存保护

内核通过页表和权限检查确保进程无法越界访问。

页表权限机制

ARM64 四级页表 (4KB page):
  PGD → PUD → PMD → PTE

PTE 权限位：
  AP[2:1]  → EL0 读/写权限
  UXN/PXN  → 用户/特权执行禁止
  AF       → Access Flag (必须置 1 才允许访问)

mmap/mprotect 如何生效：
  mmap(PROT_READ|PROT_WRITE)
    → vma.vm_flags = VM_READ | VM_WRITE
    → PTE: AP 允许 EL0 读写, PXN=1 (不可执行)

  mprotect(addr, len, PROT_EXEC)
    → 修改 VMA 的 vm_flags
    → change_protection() → 遍历 PTE 修改权限位
    → flush_tlb_range()

用户/内核空间隔离

ARM64 地址空间划分：
  TTBR0_EL1 → 用户页表 (EL0 可见)
  TTBR1_EL1 → 内核页表 (仅 EL1 可见)

PAN (Privileged Access Never):
  内核不能直接访问用户空间内存
  → 必须用 copy_from_user() / copy_to_user()
  → 这些函数临时禁用 PAN 或使用 LDTR/STTR

KASLR + KPTI:
  内核地址随机化 + 内核页表隔离
  → 防止侧信道攻击泄露内核布局

Buddy 分裂与合并

分配 order=2 (16KB) 时：
  1. 查 free_area[2] → 空
  2. 查 free_area[3] → 找到一个 32KB 块
  3. 取出 32KB，分裂为两个 16KB
  4. 一个放入 free_area[2]，另一个返回

释放 order=2 时：
  1. 找到 buddy (地址 XOR 0x4000)
  2. buddy 也在 free_area[2]？
     → 是: 合并为 order=3，继续向上检查
     → 否: 放入 free_area[2]

SLAB/SLUB 分配器 (kmalloc):
  kmalloc(64)  → 从 size-64 cache 分配
  kmalloc(256) → 从 size-256 cache 分配
  kmalloc(8192)→ 直接走 buddy (order=1)

  SLUB 在 buddy 之上提供小对象的高效分配
  → 减少碎片，提高缓存局部性

// get_unused_fd_flags() → alloc_fd() 核心逻辑

static int alloc_fd(unsigned start, unsigned end, unsigned flags)
{
    struct files_struct *files = current->files;
    struct fdtable *fdt;

    spin_lock(&files->file_lock);
    fdt = files_fdtable(files);

    // 1. 用 full_fds_bits 跳过已满的 BITS_PER_LONG 区间
    // 2. 用 open_fds 位图在目标区间找第一个 0 bit
    fd = find_next_zero_bit(fdt->open_fds, end, start);

    // 3. 如果需要扩容
    if (fd >= fdt->max_fds)
        expand_fdtable(files, fdt, fd);

    // 4. 标记为已使用
    __set_open_fd(fd, fdt);
    if (flags & O_CLOEXEC)
        __set_close_on_exec(fd, fdt);

    spin_unlock(&files->file_lock);
    return fd;
}

2.2.3 进程创建 — fork() → copy_process()

linux-src/kernel/fork.c

fork() / clone() → kernel_clone() :2764 → copy_process() :2137 → wake_up_new_task() 
struct task_struct *copy_process(struct pid *pid, int trace,
                                int node, struct kernel_clone_args *args)
{
    // ① 分配并复制 task_struct + 内核栈
    p = dup_task_struct(current, node);

    // ② 根据 clone flags 复制或共享各子系统
    retval = copy_creds(p, clone_flags);      // 凭证
    retval = copy_files(clone_flags, p);      // 文件描述符表
    retval = copy_fs(clone_flags, p);         // cwd/root/umask
    retval = copy_signal(clone_flags, p);     // 信号处理
    retval = copy_mm(clone_flags, p);         // 地址空间 (COW!)
    retval = copy_namespaces(clone_flags, p); // 命名空间
    retval = copy_thread(clone_flags, args->stack, args->tls, p); // 架构相关

    // ③ 分配 PID，加入进程树
    pid = alloc_pid(p->nsproxy->pid_ns_for_children, ...);
    p->pid = pid_nr(pid);

    // ④ 初始化调度实体
    sched_fork(clone_flags, p);

    return p;
}

CLONE 标志位决定共享还是复制

CLONE_VM      → 共享 mm_struct (线程)     vs 复制 mm_struct (进程)
CLONE_FILES   → 共享 files_struct (线程)  vs 复制 fd 表 (进程)
CLONE_FS      → 共享 fs_struct            vs 复制 cwd/root
CLONE_SIGHAND → 共享 sighand_struct       vs 复制信号处理表
CLONE_THREAD  → 同一线程组 (要求 CLONE_VM + CLONE_SIGHAND)

pthread_create() = clone(CLONE_VM|CLONE_FS|CLONE_FILES|CLONE_SIGHAND|CLONE_THREAD)
fork()           = clone(SIGCHLD)  (几乎全部复制)

总结：两个视角的统一

视角一：内核 = 事件处理器 + 数据结构

3 种事件入口：
  ① System Call (svc) → sys_call_table[nr]
  ② Interrupt (IRQ)   → gic_handle_irq → device ISR
  ③ Exception (Fault) → do_page_fault → memory management

7 个核心数据结构：
  task_struct → mm_struct → files_struct → struct file
                              ↕
                         struct sock ↔ sk_buff
                              ↕
                         struct page

视角二：内核 = 裁判 + 管家 + 翻译官

裁判 (Referee):
  CFS 调度 (vruntime + RB-tree)
  页表权限 (mmap/mprotect)
  Capabilities + Seccomp

管家 (Steward):
  Buddy + SLAB (物理内存)
  FD 位图 (文件描述符)
  copy_process() (进程)
  sk_buff 缓存 (网络缓冲)

翻译官 (Translator):
  VFS (file_operations)
  Block Layer (blk_mq_ops)
  Net Device (net_device_ops)
  Char Device (file_operations)
